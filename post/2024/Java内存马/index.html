Java内存马

1. 内存马简介

1.1 什么是内存马

Java Memory Shell：Java内存马是一种利用 Java 虚拟机（JVM）中的 内存对象 和 反射技术 来执行恶意代码的攻击手段。

与传统木马相比，Java内存马具有以下几个优势：

  1. 隐藏性更强：Java内存马不需要在受害者主机上创建任何文件，因此可以极大地减少被检测到的风险。它们完全存在于JVM内存中，很难通过常规的病毒扫描程序或防火墙来检测和拦截。
  2. 反应速度更快：Java内存马直接利用JVM中已经加载的Java类和对象，在攻击者成功上传恶意代码后即可立即执行，反应速度更快。
  3. 跨平台支持更好：由于JVM是跨平台的，因此Java内存马可以在各种操作系统上运行，包括Windows、Linux、macOS等。

1.2 内存马类型

根据内存马注入的具体技术手段分类，可分为两大类：

  • Servlet-API 型内存马

​ 通过命令执行漏洞、反序列化漏洞、已有传统 Webshell 木马等可以 RCE 执行命令的攻击前提，借助 Java 反射技术，在 JVM 中动态注册一个新的 listener、filter 或者 servlet 组件，从而实现在内存中注入可命令执行的无落地文件类的隐蔽木马。特定框架、容器的内存马原理与此类似，如 spring 的controller 内存马，tomcat 的 valve内存马。

  • Java-agent 型内存马

​ Java Agent 简单来说就是 JVM 提供的一种动态 hook class 字节码的技术，通过 Instrumentation（Java Agent API），开发者（攻击者）能够以一种无侵入的方式 （类似 Spring AOP），在 JVM 加载某个 class 之前修改其字节码的内容，或者修改已经被 JVM 加载过的 class，此技术正常情况下可被用于 Java 程序的性能监控、信息收集、问题诊断等。而 Agent 内存马的实现就是利用了这一特性，动态修改特定类的特定方法，在内存中注入恶意代码。

  • 框架型内存马

​ 除了传统的 Servlet 项目，使用 Spring 全家桶进行开发的项目越来越多，而 Spring-MVC 则是自实现了相关路由注册查找逻辑，以及使用拦截器来进行过滤，思想上与 Servlet-Filter 的设计类似。

  • 中间件型内存马

​ 在中间件的很多功能实现上，因为采用了类似 Filter-FilterChain 的职责链模式，可以被用来做内存马，由于行业对 Tomcat 的研究较多，因此大多数的技术实现和探究是针对 Tomcat 的，但其他中间件也有相当多的探究空间。

  • 其他内存马

​ 还有一些其他非常规的利用思路，可以用在内存马的实现上，例如 WebSocket 协议等。除了基于 Web 协议的内存马，可以使用各种协议作为内存马的通信途径，如 grpc、jmx、jpda 等，或封装多层协议；

2. 背景知识

2.1 反射机制

反射机制(Reflection) 是指在程序的运行状态中，可以构造任意一个类的对象，可以了解任意一个对象所属的类，可以了解任意一个类的成员变量和方法，可以调用任意一个对象的属性和方法。这种动态获取程序信息以及动态调用对象的功能称为Java语言的反射机制。

优点：

  • 灵活性：反射允许在运行时检查类的属性和方法，这使得代码更加灵活，可以动态加载类和调用方法。

  • 解耦：使用反射可以减少代码之间的耦合，使得模块更独立，便于测试和维护。

  • 框架支持：许多 Java 框架（如 Spring、Hibernate）使用反射来实现依赖注入和其他动态行为，提高了开发效率。

  • 通用性：反射可以处理任何对象，而不仅限于特定的类，这使得它可以用于通用算法的实现。

缺点：

  • 性能开销：反射的性能通常比直接调用慢，因为需要进行额外的检查和操作。

  • 安全性问题：反射可以==绕过访问控制==，可能导致安全漏洞，尤其是在处理敏感信息时。

  • 类型安全：使用反射时，类型检查是在运行时进行的，这可能导致潜在的运行时错误，而不是编译时错误。

  • 复杂性：使用反射可能使代码变得更加复杂，难以理解和维护。

反射相关的主要类:

  • java.lang.Class：代表一个类，Class 对象表示某个类加载后在堆中的对象
  • java.lang.reflect.Method：代表类的方法，Method 对象表示某个类的方法
  • java.lang.reflect.Field：代表类的成员变量，Field 对象表示某个类的成员变量
  • java.lang.reflect.Constructor：代表类的构造方法，Constructor 对象表示构造器

Java 程序在计算机中的运行可以分为以下三个主要阶段：

主要步骤：

  1. 获取 Class 对象：使用 Class.forName("类名")、类名.class 或 对象.getClass() 获取目标类的 Class 对象。
  2. 获取构造函数：通过 getDeclaredConstructor() 或 getConstructors() 获取构造函数。
  3. 创建实例：使用 newInstance() 方法创建对象实例。
  4. 获取字段和方法：使用 getDeclaredField("字段名") 和 getDeclaredMethod("方法名", 参数类型...) 获取字段和方法。
  5. 访问和调用：通过 Field.set() 和 Method.invoke() 分别访问字段和调用方法。

2.2 Java Web 三大组件

Java Web 三大组件：Servlet、Filter、Listener

2.2.1 Servlet

servlet 是一种运行服务器端的java应用程序，具有独立于平台和协议的特性，并且可以动态的生成web页面，它工作在客户端请求与服务器响应的中间层。servlet 的主要功能在于交互式地浏览和修改数据，生成动态 Web 内容。

servlet 的生命周期：

  1. 加载与实例化

实例化是在 Servlet 容器加载 Servlet 类后，通过反射创建 servlet 实例的过程。这个过程通常发生在以下两种情况下：

  • 延迟加载：默认情况下，Servlet 容器在接收到第一次客户端请求时，才会实例化该 Servlet。这种方式节省了内存和资源，避免了对不需要的 Servlet 进行实例化。

  • 预加载：如果在 web.xml 中通过 元素进行配置，Servlet 容器会在服务器启动时立即实例化该 Servlet，而不等待客户端请求。

  1. 初始化

初始化是指 Servlet 容器在实例化 Servlet 后，调用其 init() 方法进行初始化的过程。这一步是在 Servlet 被实例化后立即进行的，且仅会执行一次。

  1. 处理请求

当客户端发送请求时，容器会调用 Servlet 的 service() 方法来处理每一个请求。service() 方法根据请求的类型（GET、POST等）决定调用相应的 doGet()、doPost() 等方法。

  1. 销毁

当 Servlet 容器决定要移除 Servlet 实例时，会调用 destroy() 方法。通常在服务器关闭或 Servlet 被卸载时执行。destroy() 方法也只会被调用一次，通常用于释放资源。

2.2.2 Filter

Filter 是介于 Web 容器和 Servlet 之间的过滤器，用于过滤未到达 Servlet 的请求或者由 Servlet 生成但还未返回响应。客户端请求从 Web 容器到达 Servlet 之前，会先经过 Filter，由 Filter 对 request 的某些信息进行处理之后交给 Servlet。同样，响应从 Servlet 传回 Web 容器之前，也会被 Filter 拦截，由 Filter 对 response 进行处理之后再交给 Web 容器。

Filter 的生命周期：

  1. 初始化

当 Web 应用启动时，Servlet 容器会加载并初始化与该应用关联的 Filter，而不是在每次请求时初始化。容器会调用 Filter 的 init() 方法进行初始化。每个 Filter 实例在 Web 应用生命周期中只会初始化一次。（当 Web 容器启动并加载 Web 应用时，它会根据配置文件或注解扫描并加载所有的 Filter，然后调用每个 Filter 的 init() 方法进行初始化。这一过程只会发生一次，之后 Filter 将一直存在，直到 Web 应用被卸载或容器关闭。）

  1. 过滤操作

doFilter() 方法是 Filter 的核心方法，每当请求到达与 Filter 关联的资源（如 Servlet 或 JSP 页面）时，doFilter() 方法会被调用。doFilter() 用来拦截请求和响应，并执行预处理（即在请求传递给下一个 Filter 或目标 Servlet 之前，可以进行检查、修改或记录）或后处理操作（在 Servlet 生成响应后，Filter 可以对响应进行修改或后处理操作。）。

  1. 销毁

当 Web 应用被关闭或 Filter 被卸载时，容器会调用 Filter 的 destroy() 方法来释放资源。

2.2.3 Listener

Listener 是 Java Servlet 规范中用于监听和响应特定事件的组件。Listener 可以监听 Servlet 容器中发生的各种事件，例如 Web 应用的启动和关闭、会话的创建和销毁、属性的增加或删除等。通过 Listener，可以在应用生命周期的不同阶段执行一些特定的操作。

Listener 主要分为以下三个大类：

  • ServletContextListener ：ServletContext监听
  • HttpSessionListener ：Session监听
  • ServletRequestListener ：Request监听

由于HttpSessionListener 主要监听会话的创建和销毁，而不直接参与 HTTP 请求和响应的处理。ServletContextListener 的功能是监听整个 Web 应用的启动和关闭。它们的生命周期局限于会话和应用的启动/停止事件，无法持续处理每个 HTTP 请求。而 ServletRequestListener 的生命周期与每个请求的创建和结束相关，所以 ServletRequestListener 最适合做为内存马。

ServletRequestListener 的生命周期：

  1. 请求创建时

当一个新的 HTTP 请求到达服务器时，服务器会创建一个 ServletRequest 对象，并调用 ServletRequestListener 的 requestInitialized(ServletRequestEvent sre) 方法。

  1. 请求销毁阶段

当服务器处理完一个 HTTP 请求之后，ServletRequest 对象将会被销毁，此时服务器会调用 ServletRequestListener 的 requestDestroyed(ServletRequestEvent sre) 方法。

2.3 Tomcat 中的三个 Context

ServletContext 是 Java Servlet 规范中的一个接口，表示整个 Web 应用的上下文。每个 Web 应用在服务器启动时都会创建一个 ServletContext 对象，且只会有一个 ServletContext 对象与之对应，供该应用中的所有 Servlets、JSP、过滤器和监听器共享，用于管理应用的全局信息和资源。

ApplicationContextFacade 是对 ApplicationContext 的一个封装，因此可以看作是 ApplicationContext 的“外观”或“代理”。它并不是直接替代 ApplicationContext，而是提供了一种更安全和便捷的方式来代理访问其功能。开发者通过 ServletContext 接口与 Web 应用交互，而 ApplicationContextFacade 通过 ServletContext 提供与 ApplicationContext 的连接。因此，ApplicationContextFacade 可以看作是 Web 应用的一个“入口”，而 ApplicationContext 则是实际的上下文管理器。

StandardContext 是 Tomcat 中表示 Web 应用的核心类，它实现了 Context 接口。它负责管理 Web 应用的生命周期、Servlet 和过滤器的管理、资源访问等功能。

总结：ServletContext 接口的实现类为 ApplicationContext 类和 ApplicationContextFacade 类，其中 ApplicationContextFacade 是对 ApplicationContext 类的包装。我们对 Context 容器中各种资源进行操作时，最终调用的还是 StandardContext 中的方法，因此 StandardContext 是Tomcat中负责与底层交互的 Context。

2.4 tomcat 架构设计

Tomcat 的本质其实就是一个 WEB 服务器 + 一个 Servlet 容器，那么它必然需要处理网络的连接与 Servlet 的管理，因此，Tomcat 设计了两个核心组件来实现这两个功能，分别是连接器（Connector）和容器（Container），连接器用来处理外部网络连接，容器用来处理内部 Servlet。

一个 Tomcat 代表一个 Server 服务器，一个 Server 服务器可以包含多个 Service 服务，Tomcat 默认的 Service 服务是 Catalina，而一个 Service 服务可以包含多个连接器，因为 Tomcat 支持多种网络协议，包括 HTTP/1.1、HTTP/2、AJP 等等，一个 Service 服务还会包括一个容器，容器外部会有一层 Engine 引擎所包裹，负责与处理连接器的请求与响应，连接器与容器之间通过 ServletRequest 和 ServletResponse 对象进行交流。

一个 engine 可以对应多个 host，也就是虚拟主机，一个 host 可以对应多个 context，也就是 web 应用，一个 context 对应多个 wrapper，也就是 servlet。这个映射关系，通过 mapper 组件来关联，mapper 组件保存了 Web 应用的配置信息，容器组件与访问路径的映射关系。Host 容器的域名，Context 容器中的 web 路径，Wrapper 容器中的 servlet 映射的路径，这些配置信息是多层次的 Map。

当 Tomcat 接收到客户端请求时，首先会使用 Connector 进行解析，然后发送到 Container 进行处理。

2.5 Pipeline & Valve

当 Tomcat 接收到客户端请求时，首先会使用 Connector 进行解析，然后发送到 Container 进行处理。那么我们的消息又是怎么在四类子容器中层层传递，最终送到 Servlet 进行处理的呢？这里涉及到的机制就是 Tomcat 管道机制。

管道机制主要涉及到两个名词，Pipeline（管道）和 Valve（阀门）。如果我们把请求比作管道（Pipeline）中流动的水，那么阀门（Valve）就可以用来在管道中实现各种功能，如控制流速等。

因此通过管道机制，我们能按照需求，给在不同子容器中流通的请求添加各种不同的业务逻辑，并提前在不同子容器中完成相应的逻辑操作。个人理解就是管道与阀门的这种模式，我们可以通过调整阀门，来实现不同的业务。

Pipeline 中会有一个最基础的 Valve，这个 Valve 也被称之为 Basic，它始终位于末端（最后执行），它在业务上面的表现是封装了具体的请求处理和输出响应，分别为 StandardEngineValve，StandardHostValve，StandardContextValve，StandardWrapperValve。Pipeline 提供了 addValve 方法，可以添加新 Valve 在 basic 之前，并按照添加顺序执行。
