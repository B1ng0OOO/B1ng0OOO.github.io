1. 解读

1.1 什么是类加载？

类加载 是指将类的字节码（.class 文件）从文件系统或网络等源读取到内存中，并将其转换为 JVM 能够理解的格式的过程。类加载是 JVM 执行 Java 程序的关键步骤之一，确保类在被使用之前已经正确加载到内存中。

1.2 什么时候会发生类加载？

类加载一般在首次 主动使用类 时触发。

  1. 创建类的实例时
  2. 访问类的静态成员时（静态变量或静态方法）
  3. 调用 Class.forName() 方法显式加载类时
  4. 访问编译期不可确定的静态常量时

如果访问的是静态常量，且该常量的值需要在运行时计算，那么在首次访问时，类会被加载。

  1. 类的子类初始化时

当某个类的子类被初始化时，如果该子类依赖父类中的静态字段或静态方法，那么会首先触发父类的加载和初始化。

  1. JVM 启动时加载主类（包含 main() 方法的类）
  2. 动态代理类生成时
  3. 反射创建类实例时

[!CAUTION]

使用 Person.class 获取类的 Class 对象引用时，不会触发类的加载。

1.3 类是如何被加载的？

  1. 加载

加载阶段是类加载的第一步，在这个阶段，JVM 通过类加载器将字节码文件（.class 文件）从磁盘或网络读取到内存中，并生成一个 Class 对象，该对象代表类的元信息。查找并加载类的字节码，将类的二进制数据读入 JVM 内存，创建一个 Class 对象，用来封装该类的信息（类名、方法、属性等）。加载完成后，类会被 JVM 标记为“已加载”，但还没有进行后续的初始化和链接。

  1. 连接

连接阶段将类的字节码转换为 JVM 可以直接使用的格式，并对类的各种符号引用进行解析。连接阶段又细分为三个子阶段：

  • 验证：在验证阶段，JVM 会检查加载的字节码是否符合 JVM 规范，并确保类文件没有被篡改或破坏，保证字节码的安全性和合法性。如果验证失败，类的加载会被终止，并抛出 VerifyError 等异常。

  • 准备：准备阶段为类的静态变量分配内存，并将其初始化为默认值，而不是程序中定义的值。

  • 解析：解析阶段是将类的常量池中的符号引用（如方法、字段名等）替换为直接引用（内存地址）。符号引用是编译期生成的，而直接引用是在类加载完成后通过查找相关对象在内存中的地址得到的。

  1. 初始化

初始化阶段是类加载的最后一步。在这个阶段，类的静态变量会被赋予程序中定义的初始值，静态代码块也会被执行。这个阶段也是类的主动使用阶段，当类的静态字段或静态方法被使用时，类会在此阶段执行。

  1. 使用

在类完成加载、连接和初始化后，JVM 可以使用该类进行实例化或调用其静态成员、静态方法等。此时，类的所有静态成员和方法都可以正常使用。

  1. 卸载

当一个类不再被使用且没有被引用时，JVM 会通过垃圾回收机制将其从内存中卸载。类的卸载是由 JVM 自动完成的，一般不会手动触发。类的卸载过程释放类加载器和类的 Class 对象所占用的内存。

1.4 类加载器

类加载器是 Java 用来动态加载 .class 文件的组件，它负责从文件系统、网络等不同来源中加载类的字节码，并将它们转换为 JVM 可以使用的 Class 对象。每个类加载器都是 ClassLoader 类的一个实例，Java 提供了不同类型的类加载器用于加载不同的类。

  1. 启动类加载器（Bootstrap ClassLoader）

启动类加载器是最顶层的类加载器，由 C++ 语言实现，负责加载 Java 核心类库（如 rt.jar）中的类，如 java.lang.String、java.util 等。启动类加载器无法通过 Java 代码直接访问。

  1. 扩展类加载器（Extension ClassLoader）

扩展类加载器负责加载 JRE 的扩展类库，通常是 jre/lib/ext 目录下的类。扩展类加载器是由 Java 实现的，可以通过 ClassLoader 类的 getParent() 方法获取。

  1. 应用类加载器（Application ClassLoader）

应用类加载器负责加载应用程序类路径下的类（通常是 classpath 下的类），这是 Java 程序中最常用的类加载器。当运行一个 Java 程序时，大多数应用类都是由该类加载器加载的。

  1. 自定义类加载器（Custom ClassLoader）

Java 允许开发者创建自定义的类加载器，通过继承 ClassLoader 类并覆盖 findClass() 方法，可以定制类的加载过程。

1.5 双亲委派机制

双亲委派机制是类加载器加载类时的一种约定，即当一个类加载器收到类加载请求时，它会先将请求委派给它的父类加载器进行加载，只有当父类加载器无法加载该类时，子加载器才会尝试自己加载。这种机制确保了核心类库（如 java.lang.Object 等）不会被自定义的类加载器重复加载或覆盖，从而确保 Java 核心类库的安全性和一致性，防止核心类库被恶意篡改或被覆盖。

双亲委派机制的工作流程：

  1. 类加载器收到类加载请求后，先将该请求委派给父类加载器。
  2. 如果父类加载器能够成功加载该类，则直接使用父类加载器加载的类。
  3. 如果父类加载器无法找到该类，子类加载器才会尝试自己加载。

  • 应用类加载器（Application ClassLoader）首先收到请求加载 Person 类。

  • 它会将加载请求委派给它的父类加载器——扩展类加载器。

  • 扩展类加载器又将请求继续委派给它的父类，即启动类加载器。

  • 启动类加载器尝试加载 Person 类，但找不到该类，于是返回控制给扩展类加载器，扩展类加载器同样找不到，最终由应用类加载器尝试加载，成功加载 Person 类。

这种机制确保了像 java.lang.Object 这样的类永远由启动类加载器加载，即使在自定义的类加载器中存在同名的类，也不会破坏核心类的加载。

1.6 为什么需要双亲委派机制？

双亲委派机制的主要目的是确保 Java 核心类库的安全性和一致性，防止核心类库被恶意篡改或被覆盖。

  • 避免重复加载：通过双亲委派机制，确保 java.lang.Object 等核心类库只会被启动类加载器加载一次，不会因为应用程序中的自定义类加载器而重复加载或覆盖。

  • 防止安全漏洞：如果没有双亲委派机制，用户可以在自定义类加载器中定义与核心类库同名的类，可能会导致核心类库被错误地加载，从而引发安全问题。
