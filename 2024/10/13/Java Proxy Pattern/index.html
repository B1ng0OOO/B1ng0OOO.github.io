Java Proxy Pattern
2024-10-15 20:51:38 #Java

1. 解读

代理模式（Proxy Pattern）是一种结构型设计模式，它允许一个对象（代理）代表另一个对象（真实对象）来控制对该对象的访问。

举一个很贴切的例子，例如：在银行里，我们可以通过柜台（真实对象）进行各种交易，但也可以通过 ATM 机（代理）进行取款、查询余额等操作。ATM 机在这里面就充当了代理的角色。它代表银行为我们提供服务，我们不需要直接与银行的柜台工作人员互动就可以完成各种操作。ATM 机控制着我们对银行账户的访问，确保所有操作都是安全的。

2. 分类

Java 中的代理模式主要有两种，分别为 静态代理 和 动态代理（ JDK 动态代理 与 CGLIB 动态代理 ）。

Java 静态代理模式 主要通过在编译时创建代理类，实现对目标对象的功能增强。静态代理模式的核心思想是代理类和目标类都实现相同的接口，代理类在调用目标类的方法前后添加额外的逻辑处理。

Java 动态代理模式 是一种在运行时动态生成代理对象，并在不修改原有类代码的情况下增强功能的设计模式。与静态代理不同，动态代理不需要为每个接口或类编写单独的代理类，而是通过反射机制在运行时创建代理对象。

3. 静态代理

静态代理的特点：

  • 编译时确定代理类：代理类和目标类在编译时就已经确定。

  • 增强目标对象的功能：代理类可以在调用目标对象的方法前后添加额外的逻辑。

  • 接口依赖：静态代理通常要求目标类和代理类实现相同的接口。

使用 银行柜台（真实对象） 和 ATM 机（代理） 来实现静态代理的例子：

  • 定义银行服务的接口

BankService 接口定义了银行服务的两个核心操作：取款 (withdraw) 和查询余额 (checkBalance)

  • 实现银行柜台类（真实对象）

BankCounter 类是实际提供服务的对象，它实现了 BankService 接口，提供了取款和查询余额的具体实现。

  • 实现 ATM 机 （代理对象）

ATM 类同样实现了 BankService 接口，但它本身并不直接处理取款或查询余额操作，而是作为代理，调用 BankCounter 中的相应方法。

  • 使用静态代理

客户端通过 ATM 对象来访问银行服务，而不是直接操作 BankCounter。这其中 ATM 就代理了客户与银行柜台的交互过程。

静态代理的优点:

  1. 符合单一职责原则：代理对象负责控制对目标对象的访问，目标对象专注于业务逻辑的处理。
  2. 增强功能：不修改目标对象的情况下，增加额外功能。

静态代理的缺点:

  1. 代码冗余：每个目标对象都需要创建一个对应的代理类。如果有多个接口或多个类需要代理，则需要手动编写大量代理类。
  2. 扩展性差：如果接口发生变更，代理类和目标类都需要修改，维护成本高。

JDK 动态代理 是 Java 中一种实现代理模式的方式，它要求目标对象必须实现一个或多个接口。JDK 动态代理在运行时生成代理类，并可以在代理类的方法中添加额外的逻辑。在学习实现 JDK 动态代理实现动态创建代理对象之前，先了解一下 Proxy 类和 InvocationHandle 接口这两个 JDK 动态代理的核心。

  • java.lang.reflect.Proxy 类：用于创建代理对象。

  • java.lang.reflect.InvocationHandler 接口：用于定义代理逻辑的处理器接口，处理方法调用的逻辑。

JDK 动态代理的实现步骤：

  • 定义接口：目标对象和代理对象都要实现的接口。
  • 实现目标类：目标对象类实现接口，目标类必须实现一个或多个接口，处理具体的业务逻辑。
  • 使用 Proxy 创建代理对象：在运行时生成代理对象，并将方法调用委托给 InvocationHandler。
  • 实现 InvocationHandler：用于处理目标对象的方法调用。

4. Proxy 类

Proxy 类是 java.lang.reflect 包的一部分，专门用于创建动态代理类。它是实现 Java 动态代理的核心类，能够在运行时为接口生成代理对象。这些代理对象在调用方法时会被委托给 InvocationHandler，通过 InvocationHandler 执行方法调用逻辑。

4.1 newProxyInstance()

newProxyInstance() 是 Proxy 类最常用的方法，用于创建动态代理对象。这个方法动态生成一个实现了指定接口的代理类，并返回代理类的实例。

参数：

  • ClassLoader loader：代理类使用的类加载器。通常使用目标类的类加载器或者当前类的类加载器，代理类的加载器必须与目标类的加载器相同或可见。一般我们的目标类与我们的代理类生成器的类都是在同一个项目中编写的类，它们通常都会使用同一个应用类加载器，所以这里 ClassLoader loader 参数的值为 this.getClass().getClassLoader() 就好了。

  • Class<?>[] interfaces：代理类要实现的接口数组。代理对象会实现这些接口并拦截接口中的方法调用。

  • InvocationHandler h：一个实现了 InvocationHandler 接口的处理器对象，它定义了代理对象在方法调用时的行为逻辑。当调用代理对象的方法时，InvocationHandler 的 invoke 方法将被调用。

返回值：返回一个 动态生成的代理对象。该代理对象实现了传入的接口，并且其方法调用会被委托给提供的 InvocationHandler 来处理。

5. InvocationHandler 接口

InvocationHandler 接口定义：

InvocationHandler 是 Java 中动态代理机制的核心接口，它定义了代理类如何处理方法调用的逻辑。当代理对象的方法被调用时，调用会被转发到 InvocationHandler 接口的 invoke 方法中，我们可以在 invoke 方法中定义自定义的操作。

5.1 invoke()

参数：

  • proxy：动态代理对象。虽然不常用，但可以通过这个参数访问代理对象本身。

  • method：被调用的方法对象。代表代理对象当前调用的方法。

  • args：调用方法时传入的参数数组。如果没有参数，args 为 null。

返回值：返回值是被代理方法的返回值。如果代理方法没有返回值（即 void），则 invoke 方法返回 null。

主要研究如何通过 Proxy 类和 InvocationHandle 接口在程序运行时实现动态创建代理对象。

  • 实现 JDK 动态代理的基础功能

首先通过 setTarget() 方法传递一个目标对象，这个目标对象是我们想要代理的真实对象。再通过 getProxy() 方法，使用 Proxy.newProxyInstance() 创建一个代理对象，这个代理对象实现了和目标对象相同的接口。当我们调用代理对象的方法时，就会触发 invoke() 方法，代理对象不会直接执行目标对象的方法，而是先经过 invoke()，我们可以在 invoke() 里添加额外的逻辑。invoke() 方法内部会使用反射，调用目标对象的真实方法 (method.invoke(target, args))，并返回结果。

  • 使用 JDK 动态代理

优点：只需依赖 JDK 自带的 java.lang.reflect.Proxy 类，不需要额外的库；

缺点：只能代理接口，不能代理类本身。

7. CGLIB 动态代理

CGLIB 动态代理 是 Java 中一种实现代理模式的方式，它基于底层的字节码操作，通过继承方式为目标类生成代理对象。与 JDK 动态代理不同，CGLIB 可以代理没有实现接口的类，因为它直接生成目标类的子类，并重写其方法进行增强。

特点：

  • 不需要接口：与 JDK 动态代理要求目标类实现接口不同，CGLIB 可以为没有实现接口的类创建代理。
  • 基于继承：CGLIB 通过创建目标类的子类来生成代理类，因此目标类不能是 final，否则无法生成代理类。
  • 性能较高：相比于 JDK 动态代理，CGLIB 在处理大量方法调用时通常性能较高，因为它基于字节码操作。

优点：可以代理没有实现接口的类，灵活性更高；

缺点：需要依赖 CGLIB 库，创建代理对象的开销相对较大。目标类不能是 final 因为 CGLIB 是通过继承生成代理对象的，目标类如果被 final 修饰，CGLIB 就无法生成代理。方法不能是 final，如果目标类中的方法被 final 修饰，CGLIB 无法对这些方法进行代理。

8. JDK 与 CGLIB 动态代理区别

8.1 是否必须实现接口

由于 JDK 动态代理生成的代理类继承了 java.lang.reflect.Proxy，而且 Java 是单继承的，如果不通过实现接口的形式，无法对类进行扩展。所以 JDK 动态代理只能代理实现了接口的类。如果目标类没有实现接口，JDK 动态代理就无法使用。

而 CGLIB 动态代理生成代理类实际上通过生成目标类的子类来实现的，因此不依赖接口。

8.2 代理的生成机制不同

JDK 动态代理是通过 java.lang.reflect.Proxy 类来生成代理类的。JDK 动态代理利用了 Java 自带的 ProxyGenerator 类，这个类可以根据接口生成代理类的字节码文件，并在运行时加载到 JVM 中。

而 CGLIB 动态代理则是使用 ASM 框架生成字节码文件。ASM 是一个底层的字节码操作库，它允许直接修改和生成 Java 字节码。CGLIB 基于 ASM 框架，生成代理类时直接生成字节码，代理类会继承目标类并重写非 final 的方法。

8.3 final 类和 final 方法的限制

JDK 动态代理是基于接口对被代理类进行代理，代理类的方法调用是通过反射实现的，所以方法即便是被 final 修饰，也不影响代理行为，不影响其通过反射被访问和调用‌。

而 CGLIB 动态代理是基于类对被代理类进行代理，通过继承生成被代理类的子类来实现，final 类无法被继承和 final 方法无法被重写，所以 CGLIB 动态代理无法对 final 类和 final 方法进行代理。

8.4 代理方法调用方式的不同

JDK 动态代理中当代理对象的方法被调用时，会先将方法调用转发给 InvocationHandler 接口中的 invoke() 方法，再由 invoke() 方法使用反射来调用目标对象的实际方法。

而 CGLIB 动态代理中代理对象的方法被调用时，CGLIB 将拦截调用并将其转发给 MethodInterceptor 接口的 intercept() 方法。然后 CGLIB 会为代理类生成一个索引类，用于对目标方法进行索引和快速查找。通过索引类的 getIndex() 方法，可以更快地找到要调用的目标方法。在获取到方法索引后，CGLIB 会通过索引类的 invoke() 方法来执行对应的目标方法。在 invoke() 完成目标方法的调用后，代理类将最终接收目标类的方法调用结果。代理类可以选择对结果进行处理，或者直接返回给调用者。最终，通过 MethodProxy.invokeSuper()，代理类调用了被代理类（目标类）的实际方法。

8.5 自动生成类的数量

JDK 动态代理只会生成一个代理类，该类实现了目标对象所实现的接口，并在代理类的 invoke() 方法中对目标方法进行拦截。

而 CGLIB 动态代理相比 JDK 动态代理要复杂得多，因为 CGLIB 会生成 多个类来辅助代理的实现。

  1. 代理类：CGLIB 生成的代理类是目标类的子类。它通过继承目标类并重写非 final 方法来实现代理功能。
  2. 代理类的索引类（FastClass）：为了提高方法调用的效率，CGLIB 会为代理类生成一个索引类。这个类使用 FastClass 机制来加速方法调用的查找过程，避免了每次调用都需要使用反射。索引类会为代理类中的方法生成一个索引值，通过这个索引值来调用方法，从而避免直接使用 Method.invoke()。
  3. 被代理类的索引类（FastClass）：CGLIB 也会为目标类（即被代理类）生成一个 FastClass，它的作用与代理类的索引类类似。该类用于在代理类中快速定位目标类的方法，以提高性能。

这些索引类可以通过方法索引直接调用目标方法，避免了反射带来的性能开销。
