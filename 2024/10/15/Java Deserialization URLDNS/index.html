URLDNS链

URLDNS 链是一种基于 Java 反序列化漏洞的攻击方式，常用于实现 DNS 查询外带（exfiltration）等攻击目的。在 URLDNS 链中，攻击者通过反序列化时自动触发 DNS 查询来泄露信息或引发其他攻击行为。

URLDNS 链的入口类为 HashMap，执行类为 URLStreamHandler 的 hashCode()。

1. Gadget Chain 分析

HashMap，Url 两个类都实现了 Serializable 接口。

在 HashMap 反序列化时，HashMap 会通过 readObject() 方法恢复其内部数据结构。

readObject() 方法会调用 HashMap 的 putVal() 方法，putVal() 方法的参数中又调用了 HashMap 的 hash() 方法，hash() 方法中的参数 key 为 HashMap 对象中的键，HashMap 的 hash() 方法又会调用 key.hashCode()。

而 URL 类中也存在 hashCode() 方法，如果传入的 key 为 URL 对象，则反序列化时会调用 URL 对象的 hashCode() 方法。

如果调用了 URL 对象的 hashCode() 方法，则会调用 handler.hashCode(this)，其中 handler 为URLStreamHandler 对象，所以最终会调用 URLStreamHandler 对象的 hashCode() 方法。

在 URLStreamHandler 对象的 hashCode() 中会调用 getHostAddress(u)，进而发起 DNS 请求。

2. URLDNS CHAIN 实现

URL 的 hashCode 缓存机制：

  • URL 类有一个私有字段 hashCode，初始值为 -1，表示未计算过 hashCode。
  • 当调用 hashCode() 方法时，会通过 DNS 解析主机名，并将计算出的值存储到 hashCode 字段中，以便后续使用时避免再次计算。
  • 在 HashMap 中，put() 方法会调用键的 hashCode()，因此在 HashMap 中插入 URL 对象时，URL.hashCode() 被调用，导致 URL 对象在序列化前已经计算了 hashCode，DNS 请求也已经触发。

反序列化时的行为：

  • 由于 URL.hashCode 的值已被缓存，反序列化后，URL.hashCode() 将不会再次调用 DNS 请求，因为 hashCode 已不再是 -1。

解决方案步骤：

  1. 在调用 HashMap.put() 之前修改 hashCode：
    • 使用反射将 URL 对象的 hashCode 字段修改为一个非 -1 的值，以避免在 put() 时进行 DNS 请求。
  2. 在调用 HashMap.put() 之后将 hashCode 重置为 -1：
    • 再次使用反射将 URL 对象的 hashCode 字段重置为 -1，以确保 hashCode 字段在序列化时为未计算状态。

这样，在反序列化时，URL.hashCode() 会再次被调用，并触发新的 DNS 请求。

3. 避免 URLDNS 反序列化链

避免在 HashMap 中使用 URL 作为键：

  • 尽量避免将 URL 对象作为 HashMap 的键。如果必须使用 URL，可以使用其字符串形式或其他能避免网络请求的替代方案。

自定义 HashMap 的反序列化逻辑：

  • 通过自定义 HashMap 的反序列化过程（即覆盖 readObject()）来避免调用 URL 的 hashCode() 方法。例如，可以在反序列化过程中延迟计算 hashCode() 或跳过某些可能触发网络请求的逻辑。

**使用 transient 或 writeReplace()**：

  • 将容易触发 DNS 请求的对象（如 URL）标记为 transient，或者使用 writeReplace() 和 readResolve() 方法来定制序列化和反序列化行为，确保在反序列化时不会进行网络操作。

安全反序列化技术：

  • 使用安全的反序列化框架或方法（如基于白名单的反序列化）来避免潜在的不安全反序列化操作，防止不可信数据引发恶意操作。
