1. 序列化和反序列化的区别

1.1 序列化

1.1.1 定义

Serialization（序列化）是指将数据结构或对象状态转换成可以存储或传输的格式（字节序列）的过程。这个格式通常是和平台无关的，比如 JSON，XML，二进制格式等，以便可以在不同的系统或环境中进行交换。序列化后的数据可以存储在文件中，或者通过网络发送给其他系统。

1.1.2 作用

  • 数据持久化：序列化可以将对象转换为字节流，然后将其存储在文件或数据库中，以便在程序关闭或系统重启后能够重新加载这些数据。
  • 网络传输：序列化使对象能够通过网络在不同的 JVM（Java 虚拟机）之间传输。通过序列化，可以将对象转换为字节流并发送到远程系统，接收方可以通过反序列化恢复原始对象。
  • 对象深拷贝：通过序列化将对象写入内存，然后反序列化创建一个全新的对象，可以实现对象的深拷贝。（需要完全独立的对象副本，而不是简单的引用复制时。）

1.2 反序列化

1.2.1 定义

Deserialization（反序列化）是序列化的逆过程，即将序列化后的字节序列数据（如 JSON，XML，二进制格式等）转换回原始的数据结构或对象状态。这个过程允许对象在需要时重新构建对象，并使用其原始数据。

当一个对象被反序列化时，Java 会尝试根据字节流重新构造这个对象。这个过程通常如下：

  1. 创建一个新实例：Java 会通过调用反射 API 来创建一个该类的实例。
  2. 初始化字段：通过读取存储在字节流中的数据，将字段的值设置为存储的值。

[!CAUTION]

注意：在 Java 反序列化过程中，不需要调用类的构造方法（包括无参构造方法）来创建类的实例。然而，如果类的父类没有实现 Serializable 接口，则需要父类提供一个无参构造方法，不然会导致反序列化失败。

1.2.2 作用

  • 数据恢复：对象的状态可以在不同时间点通过序列化保存，之后通过反序列化恢复。以恢复程序运行前的状态。
  • 数据接收：在网络通信中，接收到的序列化数据需要被反序列化成对象，以便在本地系统中使用。

2.序列化的信息

在 Java 序列化过程中，将对象转换为字节流的过程中，序列化不仅仅是将对象的字段数据进行存储，还包括了额外的信息。

2.1 类的元数据信息

  • 全限定名：保存类的完整名称，包含包路径。
  • serialVersionUID：类的唯一标识符，反序列化时用于验证类版本的一致性。

2.2 字段信息

  • 非 transient 和非 static 字段：在序列化时，Java 会保存对象中的所有非 transient 和非 static 字段。基本数据类型（如 int、double）直接保存它们的值，引用数据类型（如 String、数组、集合、自定义类）则递归保存其内容。自定义类也需要实现 Serializable 接口，不然对象无法实例化。
  • static 字段：因为 static 字段属于类本身，而不是某个具体对象，序列化时不会保存 static 字段的值。
  • transient 字段：transient 修饰的字段不会被序列化。反序列化时，这些字段会被设置为默认值（如 null、0 等）。若自定义了对象的序列化过程，可以选择在 writeObject() 方法中手动序列化 transient 字段。在反序列化时，使用 readObject() 方法恢复这些字段的值。

2.3 自定义序列化与反序列化方法

如果类定义了 writeObject(ObjectOutputStream oos) 和 readObject(ObjectInputStream ois) 方法，序列化和反序列化时会优先调用这些方法，用于自定义序列化的逻辑。

  • writeObject：序列化时，writeObject 方法会被调用，可以控制如何序列化对象的字段，并可以执行额外的操作，比如对敏感数据进行加密。

  • readObject：反序列化时，readObject 方法会被调用，用于自定义恢复字段的过程。可以通过该方法验证数据或执行其他自定义操作。

2.4 继承层次中的字段

序列化不仅会序列化子类的字段，还会序列化所有父类的非 transient 字段。如果父类实现了 Serializable 接口，那么父类的字段也会被正常序列化。

  • 父类实现了 Serializable：如果父类也实现了 Serializable 接口，父类的非 transient 字段会正常序列化并在反序列化时恢复。父类的 transient 字段在自定义序列化方法中是也可以被自定义序列化的，并在反序列化过程中恢复。
  • 父类没有实现 Serializable：如果父类没有实现 Serializable，那么父类的字段不会被序列化。反序列化时，父类部分会通过调用父类的无参构造方法来初始化。

2.5 外部化的 Externalizable 接口

如果类实现了 Externalizable 接口，而不是 Serializable 接口，序列化时会调用 writeExternal(ObjectOutput out) 方法，反序列化时会调用 readExternal(ObjectInput in) 方法。与 Serializable 不同，Externalizable 不会自动序列化类的字段，必须在 writeExternal 和 readExternal 方法中手动指定需要序列化的内容。

2.6 序列化标记信息

序列化流还包含一些辅助的标记信息，用于帮助解析字节流。这些标记包括：

  • 对象的起始和结束标志。
  • null 对象的标志（如果字段为 null，则会记录 null 标志）。
  • 递归对象的引用标识（处理循环引用）。

这些标志信息确保反序列化时能够正确地解析和恢复对象结构。

3. Java 序列化与反序列化方式

[!CAUTION]

注意：

  • JDK 自带的序列化方式一般不会用 ，因为性能较差，序列化的数据量大，序列化和反序列化效率低并且序列化的过程不可控，容易引发安全问题。对于追求性能和效率的应用场景，通常会优先选择二进制序列化协议，如 Kryo、Protobuf、Hessian 等。而 JSON、XML 这样的文本序列化方案虽然具有可读性，但由于性能限制，一般只在一些轻量级场景或配置文件处理时使用。
  • 序列化和反序列化过程中都是严格按照顺序进行处理的。当对象被序列化时，所有字段（或自定义序列化方法中的字段）按照代码中的顺序依次写入输出流。写入的顺序和调用的顺序相同。反序列化时，必须严格按照序列化时的顺序依次读取。每个 read 操作必须与序列化时的 write 操作顺序相匹配。否则会导致数据错误或抛出异常。

3.1 实现 Serializable 接口

使用 Java 自带的序列化机制，主要通过 Serializable 接口实现。该方式是 Java 默认的序列化方式。Serialization 接口是一个空接口。

  • 实现方式：对象需要实现 java.io.Serializable 接口，使用 ObjectOutputStream 进行序列化，将对象转换为字节流。反序列化时使用 ObjectInputStream 从字节流还原对象。
  • 优点：Java 提供内置支持，使用简单。
  • 缺点：性能较低，生成的字节流较大，不适合跨语言环境，存在安全性问题（易遭受反序列化攻击）。

3.2 实现 Externalizable 接口

通过实现 java.io.Externalizable 接口，开发者可以控制对象的序列化和反序列化过程。

  • 实现方式：实现 Externalizable 接口，并覆盖 writeExternal() 和 readExternal() 方法。
  • 优点：序列化和反序列化的过程完全由开发者控制，可以优化性能。
  • 缺点：编程复杂，开发者需要手动处理每个字段的序列化。

3.3 使用 JSON 序列化库

使用 JSON 作为序列化格式，适合于跨语言的数据传输。常用的库有 Jackson 和 Gson。

  • 实现方式：使用 Jackson 或 Gson 将对象转换为 JSON 字符串，反之亦然。
  • 优点：轻量、易于阅读和调试，跨语言支持好。
  • 缺点：性能可能不如原生序列化；不适合传递复杂对象。

3.3.1 Gson

Gson 是由 Google 提供的 Java 库，可以轻松地将 Java 对象与 JSON 进行相互转换。

3.3.2 Jackson

Jackson 是另一个流行的 JSON 序列化库，功能非常强大，广泛应用于各种 Java 项目中。

3.4 使用 XML 序列化库

在 Java 中，进行 XML 序列化和反序列化通常使用 JAXB（Java Architecture for XML Binding），它可以将 Java 对象转换为 XML，或者从 XML 转换为 Java 对象。

3.5 使用 Kryo 序列化库

Kryo 是一个灵活且高效的序列化库，旨在快速和高效地将 Java 对象序列化为二进制数据，或者从二进制数据反序列化为 Java 对象，适合需要高性能序列化的 Java 应用。通过注册类和高效的字节处理，它能显著提高数据传输效率，尤其适用于大数据量和复杂对象的场景。

工作原理：

  • 对象注册：在使用 Kryo 序列化之前，需要注册要序列化的类。注册的类可以通过 Kryo 库内置的标识符进行快速查找。

  • 序列化过程：将对象的类型信息和字段值转换为字节流。序列化过程中会使用 ID 来代替对象的完整类型名称，从而减少数据传输量。

  • 反序列化过程：读取字节流并重构对象。通过已注册的类和字段信息，恢复对象的状态。

特点：

  • 速度快：Kryo 的序列化速度相对于 Java 的原生序列化要快数倍，尤其是在处理大量数据时。
  • 节省内存：Kryo使用的序列化格式更为紧凑，能有效减少序列化后的数据大小，节省内存。
  • 支持循环引用：Kryo 能够处理对象之间的循环引用，这在复杂对象图中尤其重要。
  • 支持版本控制：Kryo 提供了版本控制功能，允许开发者在序列化和反序列化时处理不同版本的类。

3.6 使用 Hessian 序列化库

Hessian 是一个高效、跨语言的二进制序列化协议，适用于需要快速数据传输的分布式系统和 RPC 应用。通过其简单的 API 和高效的序列化机制，Hessian 能够在多种编程语言之间有效地进行数据交换。

工作原理：

  • 数据编码：将对象的数据编码为二进制格式，这个过程包括将对象的类型、字段和数据值转换为一系列字节。

  • 数据传输：通过网络将编码后的数据发送到目标系统。

  • 数据解码：在接收端，将接收到的二进制数据解码为原始对象，重建其状态。

特点：

  • 简单易用：Hessian 提供简单的 API，方便用户进行序列化和反序列化操作。
  • 紧凑的格式：Hessian 使用二进制格式，能够有效减少传输数据的大小，从而提高网络传输效率。
  • 支持复杂数据结构：可以序列化复杂对象、数组、集合和 Map 等数据类型。
  • 版本兼容性：Hessian 允许用户在序列化和反序列化过程中处理版本变化。

缺点：由于是二进制协议，不如 JSON 或 XML 可读性高，调试时可能不太方便。在一些复杂的序列化场景中可能需要手动处理对象的版本控制。尽管较为快速，但在某些场景下可能不如其他高性能序列化库（如 Kryo）快。

4. static & transient

4.1 static 关键字的限制

static 变量是类级别的变量，它们在内存中只存在一份，与类本身相关，而不是与对象实例相关，因此它们不会被序列化。当序列化一个对象时，序列化过程只会保存该对象实例的状态，而不会保存类的状态，因此无论使用哪种序列化方式，static 变量都不会包含在对象的序列化状态中。

4.2 transient 关键字的限制

transient 关键字用于表示某个变量在序列化时应该被忽略，序列化时不会保存其状态。它适用于一些敏感信息或者不可序列化的资源。该关键字主要用于保护某些不需要或不希望序列化的字段，如敏感信息（密码）、不重要的临时状态、或者不能被序列化的资源（如文件句柄、线程对象等）。当一个对象被反序列化时，transient 变量将会被设为其默认值（对于对象是 null，对于基本数据类型是 0 或 false）。

4.3 transient 例外情况

transient 字段在某些特殊情况下可以被序列化，例如使用 自定义序列化 方法（通过 writeObject() 和 readObject()），可以选择在 writeObject() 方法中手动序列化 transient 字段。在反序列化时，使用 readObject() 方法恢复这些字段的值。

5. serialVersionUID

serialVersionUID 是 Java 序列化机制中用于标识类版本的一种唯一标识符，是特殊的静态字段。它用于在序列化和反序列化时确保版本的兼容性。当一个类被序列化为字节流时，serialVersionUID 会作为序列化数据的一部分存储下来。反序列化时，Java 会将存储的 serialVersionUID 与当前类的 serialVersionUID 进行比较，如果不匹配，就会抛出 InvalidClassException，表示反序列化时对象的类与序列化时不兼容。

5.1 特殊性

  • static 关键字：静态字段是属于类的，而不是属于对象的。因此，通常情况下，static 字段不会在对象序列化过程中被包含。也就是说，当你序列化一个对象时，所有 static 变量不会作为对象状态的一部分被序列化。
  • serialVersionUID 是类的元数据：Java 序列化机制使用 serialVersionUID 来表示类的版本信息。虽然它是 static 字段，但它并不需要像普通对象字段一样进行序列化。它是在序列化和反序列化过程中，隐式地由 Java 机制处理的，存储在序列化流中以确保版本一致性。Java 在序列化对象时，会将类的元数据（包括 serialVersionUID、类名等）写入到序列化流中。因此，serialVersionUID 是作为类描述信息存储在序列化流中的，而不是作为实例字段存储。

5.2 序列化过程

  • 当一个对象被序列化时，Java 首先会把该对象所属类的相关元数据写入序列化流。这些元数据包括类名、类的结构信息（如字段和方法）以及 serialVersionUID。
  • Java 序列化系统会读取这个 serialVersionUID，用它来确认反序列化时，当前类的版本和序列化时类的版本是否一致。

6. 反序列化漏洞

6.1 符合条件

  1. 共同条件：类需要实现 Serializable 接口。
  2. 入口类 source：通过重写 readObject 调用常见的调用链，并且参数类型宽泛，最好利用 JDK 自带的调用链。
  3. 调用链（gadget chain）：调用链通过 JDK 自带类，利用反序列化时的调用特性触发漏洞。
  4. 执行类（sink）：通过反序列化触发远程代码执行（RCE）、服务器端请求伪造（SSRF）或写文件等恶意操作。
